# 一青友人 · 运营后台说明文档

> 版本：v1.0 · 单文件运营后台
> 日期：2026-05-31
> 文件：`admin.html`（单文件 SPA，约 1030 行）

---

## 一、版本定位

| 项目 | 内容 |
|---|---|
| 名称 | 一青友人 · 运营后台 |
| 终端 | 桌面浏览器（建议 1280 宽以上） |
| 用途 | 早期冷启动陪聊：客服扮演女性 + AI 辅助回复 + 自动推送打招呼 |
| 与前端关系 | 通过 `localStorage('yqyr_store')` 与 `app.html` 共享数据 |
| 数据形态 | 无后端，所有数据存于浏览器 localStorage |
| 后续演进 | 接入真实后端后，仅需把 `ops.*` 与 PushWorker 改造为 REST 调用 |

> **使用场景**：男用户在 `app.html` 注册后，客服在 `admin.html` 看到对话，扮演假女性回复；同时系统按时间段自动给男用户推送打招呼，提高活跃度。

---

## 二、核心解决的业务问题

```
冷启动期 → 真女用户稀少 → 男用户匹配 0 反馈 → 流失
              ↓
       客服扮演女性陪聊（合规边界内）
              ↓
       男用户感觉"有人聊" → 付费聊天 / 送礼 / 视频
              ↓
       AI 模型双向辅助：
        - 客服端：给客服生成"女性视角"的回复参考
        - 男用户端：给男用户生成"高情商话术"参考
              ↓
       时段自动推送 → 模仿"他趣"陆续接收异性打招呼
```

## 三、技术架构

### 3.1 技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 视图 | Vue 3.4 (CDN) | 与 `app.html` 同栈，零构建 |
| 路由 | 单页内 `view` 切换 | 6 个一级视图，无 hash 路由 |
| 状态 | `reactive()` Store | 同一份 `DEFAULT_STORE` schema，双端共享 |
| 同步 | `storage` 事件 | `app.html` 写入 → 后台秒级感知 |
| AI | 智谱 GLM-4-Flash | 免费、跨域开放、稳定 |
| 样式 | 纯 CSS | 桌面后台风格（深色侧边栏 + 浅色主区） |

### 3.2 单文件结构

```
admin.html
├─ <head>
│   ├─ viewport: width=1280（强制桌面布局）
│   └─ <style> ~400 行
│       ├─ Layout（sidebar 200 + main flex）
│       ├─ 通用控件（btn / input / card / tag / av / table）
│       ├─ KPI 卡 / 漏斗图
│       ├─ 工作台 3 列布局
│       ├─ 推送规则表单
│       └─ Modal + Toast
└─ <body>
    └─ <script>
        ├─ Vue3 CDN
        ├─ DEFAULT_STORE（与 app.html 同构）
        ├─ store + 持久化 + storage 监听
        ├─ glm.chat()
        ├─ 假用户工具池（FAKE_NAME / CITY / SIGN / TAG）
        ├─ view + ui state
        ├─ computed（fakeConvs / activeConv / activeFake / totalUnread）
        ├─ ops 业务方法（addFake/delFake/batchGen/sendReply/askAI/manualPush/resetAll）
        ├─ 6 个视图组件（Dashboard/Fakes/Workbench/Push/AI/Logs）
        ├─ 根 App + 侧边栏导航
        └─ window.admin = {...}（控制台调试）
```

### 3.3 与 app.html 数据通道

```
┌────────────────────┐                ┌────────────────────┐
│      app.html      │                │     admin.html     │
│  （男用户 H5）     │                │   （运营后台）     │
└─────────┬──────────┘                └──────────┬─────────┘
          │ watch(store) 150ms 节流              │
          ▼                                      │
   ┌──────────────────────────────────────────┐  │
   │  localStorage["yqyr_store"]              │◄─┘
   │  ←── 同源浏览器同 origin ──────────────► │
   └──────────────────────────────────────────┘
          ▲                                      ▲
          │  window.addEventListener('storage')  │
          │  浅合并：对象用 Object.assign        │
          │           基本类型直接覆盖           │
          └──────────────────────────────────────┘
```

**关键约束：必须同一浏览器、同一域名打开两个 Tab**。跨设备不同步。

## 五、六大视图详解

### 5.1 数据看板 `view='dashboard'`

- **KPI 卡 ×4**：假用户数 / 已推送 / 已回复 / 付费转化
- **漏斗图**：曝光 → 打开 → 回复 → 付费（按 `adminAnalytics` 实时计算）
- **趋势提示**：基于 `pushLog` 当日/前日对比
- **快捷入口**：跳转到工作台 / 假用户管理

> **数据口径**：`pushed` = 系统/手动推送次数；`opened` 当前用占位（未来由 app.html 进入聊天页时 +1）；`replied` = 男用户回复事件 +1。

### 5.2 假用户管理 `view='fakes'`

- **顶部工具栏**：搜索框 + "新建" + "批量生成" + "重置"
- **表格列**：头像 / 昵称 / 城市 / 年龄 / 标签 / 在线 / 客服认领 / 操作
- **操作**：编辑（弹窗）/ 推送（手动）/ 删除
- **批量生成**：默认 5 个，从名字/城市/标签池随机组合
- **客服认领字段 `csOwner`**：未来多客服时区分归属（v1.0 留空即可）

> **设计要点**：假用户与真用户在 `MOCK_USERS` 中共池显示。PushWorker 触发后自动注入 `MOCK_USERS`，保证发现页/个人主页能渲染。

### 5.3 客服工作台 `view='workbench'`

**三列布局**：

| 列 | 内容 |
|---|---|
| 左 | 与假用户的对话列表（按 `fakeConvs` 过滤） |
| 中 | 选中会话的消息流（IM 风格气泡） |
| 右 | 假用户资料卡 + AI 建议 + 模板回复 |

**核心交互**：

1. 客服选会话 → 中间显示完整消息历史
2. 点 `🤖 AI 建议` → 调 `glm.chat()`，最近 6 条上下文 + 假用户人设
3. AI 返回文字直接填入输入框
4. 客服可改 / 发送 → 以 `type:'in'` 推入消息流（男用户端会出现红点）

**AI 角色对齐**：
```js
system: csConfig.systemPrompt + "\n你的角色资料：昵称{name},{age}岁,{city},性格:{persona}。"
messages: [...历史6条, 角色映射]
  // 站在女性视角：男用户(out) → role:user, 女(in/cs) → role:assistant
```

### 5.4 推送规则 `view='push'`

- **总开关**：`pushRules.enabled`
- **时段配置**：默认 3 段（早 9-11 / 中 12-14 / 晚 19-22:30），每段可配最大推送数
- **冷却**：默认 30 分钟（同一假用户对同一男用户的间隔）
- **策略**：`active`（仅活跃时段推送），未来可扩展 `aggressive` / `whale-only`
- **手动触发**：表内每行可点立即推送，用于演示/兜底

**PushWorker 在 app.html 端运行**（不在 admin.html）：
- `setInterval` 60s + 0-120s 随机
- 启动 8s 后立即首次触发（演示用）
- 命中时段、未超上限、未冷却 → 推送

### 5.5 AI 设置 `view='ai'`

- **API Key 输入**（智谱 GLM-4-Flash）
- **System Prompt** 编辑
- **快捷模板**编辑（增删行）
- **测试按钮**：直接发起一次 `glm.chat()` 试用
- **安全警告框**：⚠️ Key 暴露在前端仅适用于 demo，正式环境必须后端代理

**获取 Key 流程**：
1. 打开 https://open.bigmodel.cn/
2. 注册 → 实名 → API 密钥页 → 新建 Key
3. GLM-4-Flash 免费额度足够冷启动期使用

### 5.6 推送日志 `view='logs'`

- 表格：时间 / 假用户 / 男用户 / 时段 / 是否打开 / 是否回复
- 支持按日/按假用户筛选
- 用于复盘哪些女号转化率高、哪些时段命中率好

## 六、操作 SOP（客服日常）

### 6.1 冷启动首日

1. 后台 → AI 设置 → 录入 GLM Key
2. 假用户管理 → 批量生成 20 个
3. 推送规则 → 确认 3 个时段已开
4. 男用户首次 app.html 登录 → 8s 后会自动收到第一条打招呼
5. 工作台等待消息 → 红点提示

### 6.2 接到消息后

1. 工作台 → 点击会话
2. 看最近几条上下文 → 评估男用户意图
3. 点 `🤖 AI 建议` → 5 秒内拿到回复
4. 简单调整（去掉过分客气 / 改地名 / 加 emoji）→ 发送
5. 重要节点（送礼/视频通话邀约）→ 人工把关，不要全 AI

### 6.3 触发付费的话术钩子

| 男用户行为 | 推荐回复方向 | AI 提示词追加 |
|---|---|---|
| 反复发"在吗" | 主动反问引导他说自己 | "用一个开放式问题回他" |
| 聊到第 20+ 条 | 暗示视频通话 | "暗示想看看真人但不强求" |
| 发送礼物后 | 表达感谢 + 升温 | "感谢他的礼物并表达温柔" |
| 询问见面 | 委婉拖延 | "拒绝见面但保持兴趣" |

## 七、合规与风险

| 风险点 | 缓解措施 |
|---|---|
| 假女性陪聊 | **必须**在用户协议明示"系统可能包含运营账号"；不收高额礼物；不主动索取 |
| 诱导付费 | AI System Prompt 已设置：不主动提钱、礼物、转账 |
| 数据真实性 | 推广素材不得伪造"真实用户故事"；后台数据不可外泄 |
| AI Key 泄露 | v1.0 仅 demo；上线必须改为后端代理（前端调内部 `/api/ai/chat`，不暴露 GLM Key） |
| 未成年人 | 假用户年龄全部 ≥20；男用户实名后才进入正式付费场景 |

> **法律红线**：参考"伊对/他趣"已被多次报道的"虚拟陪聊"模式，必须避免：①保证可识别"系统账号"标签；②不诱导大额单笔付费；③客服话术不得做出"承诺线下见面"等违法陈述。

### 8.2 控制台调试

```js
// admin.html 调试
admin.store              // 全部状态
admin.ops.batchGen(10)   // 批量生成 10 个假用户
admin.ops.manualPush('f17xxx')  // 手动推送

// app.html 调试
yqyr.store

localStorage.clear(); location.reload()  // 全部重置
```

### 8.3 部署建议

- **demo 阶段**：与 `app.html` 同 origin 部署，双端共享 localStorage
- **正式阶段**：必须切换到真实后端
  - 后台改为登录态后端管理系统（RBAC）
  - 假用户改为 DB 表
  - GLM 调用走后端 `/api/ai/chat`，Key 不下发
  - 推送由后端 cron / MQ 触发，给前端推 WebSocket 消息

## 十、已知限制（v1.0）

| 限制 | 影响 | 后续 |
|---|---|---|
| 双端必须同浏览器同 origin | 客服只能本地接 demo | 接入真实后端解决 |
| GLM Key 前端可见 | 仅适合内部演示 | 后端代理 |
| `opened` 指标暂未联动 | 漏斗第二级显示偏低 | app.html 进聊天页时上报 |
| 单客服模型 | 多客服会互相覆盖 `csOwner` | 后端引入 session 锁 |
| 假用户图片仅 av-1~av-8 | 真实感不足 | 后续接 OSS 图床 |
| 推送时段未考虑节假日 | 周末/节日推送强度同工作日 | 引入日历策略 |
